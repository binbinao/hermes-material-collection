## 3. Data Flow

### 3.1 CLI Session

```
User input → HermesCLI.process_input()
  → AIAgent.run_conversation()
    → prompt_builder.build_system_prompt()
    → runtime_provider.resolve_runtime_provider()
    → API call (chat_completions / codex_responses / anthropic_messages)
    → tool_calls? → model_tools.handle_function_call() → loop
    → final response → display → save to SessionDB
```

### 3.2 Gateway Message

```
Platform event → Adapter.on_message() → MessageEvent
  → GatewayRunner._handle_message()
    → authorize user
    → resolve session key
    → create AIAgent with session history
    → AIAgent.run_conversation()
    → deliver response back through adapter
```

### 3.3 Cron Job

```
Scheduler tick → load due jobs from jobs.json
  → create fresh AIAgent (no history)
  → inject attached skills as context
  → run job prompt
  → deliver response to target platform
  → update job state and next_run
```

## 5. Major Subsystems

### 5.1 Agent Loop

- Core synchronous orchestration engine implemented as `AIAgent` in `run_agent.py`
- Responsibilities: provider selection, prompt construction, tool execution, retries, fallback, callbacks, compression, persistence
- Supports 3 API modes for different provider backends
- Reference: [Agent Loop Internals](./agent-loop.md)

### 5.2 Prompt System

Handles prompt construction and maintenance across the conversation lifecycle:

- `prompt_builder.py`: Assembles system prompt from:
  - Personality (`SOUL.md`)
  - Memory (`MEMORY.md`, `USER.md`)
  - Skills, context files (`AGENTS.md`, `.hermes.md`)
  - Tool-use guidance, model-specific instructions
- `prompt_caching.py`: Applies Anthropic cache breakpoints for prefix caching
- `context_compressor.py`: Summarizes middle conversation turns when context exceeds thresholds
- References: [Prompt Assembly](./prompt-assembly.md), [Context Compression & Prompt Caching](./context-compression-and-caching.md)

### 5.3 Provider Resolution

- Shared runtime resolver used by CLI, gateway, cron, ACP, and auxiliary calls
- Maps `(provider, model)` tuples to `(api_mode, api_key, base_url)`
- Supports 18+ providers, OAuth flows, credential pools, and alias resolution
- Reference: [Provider Runtime Resolution](./provider-runtime.md)

### 5.4 Tool System

- Central tool registry at `tools/registry.py` with 70+ registered tools across ~28 toolsets
- Tools self-register at import time
- Registry handles schema collection, dispatch, availability checking, error wrapping
- Terminal tools support 7 backends: local, Docker, SSH, Daytona, Modal, Singularity, Vercel Sandbox
- Reference: [Tools Runtime](./tools-runtime.md)

### 5.5 Session Persistence

- SQLite-based session storage with FTS5 full-text search
- Features: session lineage tracking (parent/child across compressions), per-platform isolation, atomic writes with contention handling
- Reference: [Session Storage](./session-storage.md)

### 5.6 Messaging Gateway

- Long-running process with 20 platform adapters
- Features: unified session routing, user authorization (allowlists + DM pairing), slash command dispatch, hook system, cron ticking, background maintenance
- Reference: [Gateway Internals](./gateway-internals.md)

### 5.7 Plugin System

- 3 discovery sources: `~/.hermes/plugins/` (user), `.hermes/plugins/` (project), pip entry points
- Plugins register tools, hooks, and CLI commands via a context API
- 2 specialized plugin types (single-select, only one active at a time):
  1. Memory providers (`plugins/memory/`)
  2. Context engines (`plugins/context_engine/`)
- Configured via `hermes plugins` or `config.yaml`
- References: [Plugin Guide](/docs/guides/build-a-hermes-plugin), [Memory Provider Plugin](./memory-provider-plugin.md)

### 5.8 Cron

- First-class agent tasks (not shell tasks)
- Jobs stored in JSON, support multiple schedule formats, can attach skills/scripts, deliver to any platform
- Reference: [Cron Internals](./cron-internals.md)

### 5.9 ACP Integration

- Exposes Hermes as an editor-native agent over stdio/JSON-RPC for VS Code, Zed, and JetBrains
- Reference: [ACP Internals](./acp-internals.md)

### 5.10 Trajectories

- Generates ShareGPT-format trajectories from agent sessions for training data generation
- Reference: [Trajectories & Training Format](./trajectory-format.md)

## 6. Design Principles

| Principle | Practical Implementation |
|-----------|--------------------------|
| **Prompt stability** | System prompt does not change mid-conversation. No cache-breaking mutations except explicit user actions (e.g., `/model`). |
| **Observable execution** | Every tool call is visible to the user via callbacks. Progress updates show in CLI (spinner) and gateway (chat messages). |
| **Interruptible** | API calls and tool execution can be cancelled mid-flight by user input or signals. |
| **Platform-agnostic core** | One `AIAgent` class serves CLI, gateway, ACP, batch, and API server. Platform differences live in entry points, not the agent. |
| **Loose coupling** | Optional subsystems (MCP, plugins, memory providers, RL environments) use registry patterns and `check_fn` gating, not hard dependencies. |
| **Profile isolation** | Each profile (`hermes -p <name>`) gets its own `HERMES_HOME`, config, memory, sessions, and gateway PID. Multiple profiles run concurrently. |

## 7. File Dependency Chain

Defines the import order for tool registration, which ensures all tools are auto-discovered before agent initialization:

```
tools/registry.py  (no deps — imported by all tool files)
       ↑
tools/*.py  (each calls registry.register() at import time)
       ↑
model_tools.py  (imports tools/registry + triggers tool discovery)
       ↑
run_agent.py, cli.py, batch_runner.py, environments/
```

- Key note: Any `tools/*.py` file with a top-level `registry.register()` call is auto-discovered — no manual import list is required.
