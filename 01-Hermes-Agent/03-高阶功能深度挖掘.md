## 1. Kanban 多Agent协作系统

### 1.1 核心定位

Hermes Kanban 不是又一个项目管理工具——它是 **AI Agent 的操作系统级任务调度器**。区别于 `delegate_task`（RPC 调用，fork→join），Kanban 是一个**持久化消息队列 + 状态机**，每次交接都是数据库中的一行记录，任何 Profile 或人类都可以读写。

| 维度 | delegate_task | Kanban |
|------|-------------|--------|
| 形态 | RPC 调用 (fork→join) | 持久化消息队列 + 状态机 |
| 父进程行为 | 阻塞等待子进程返回 | 创建即返回 (fire-and-forget) |
| 子进程身份 | 匿名子 Agent | 命名 Profile（带独立记忆和技能） |
| 可恢复性 | 失败 = 失败 | block → unblock → 重跑；崩溃 → 回收重派 |
| 人机协同 | 不支持 | 评论/unblock 任意时刻介入 |
| 单任务参与方 | 一次调用 = 一个子 Agent | N 个 Agent（重试/审查/跟进） |
| 审计追踪 | 上下文压缩后丢失 | SQLite 永久保存 |
| 协调方式 | 层级式（调用者→被调用者） | 对等式（任意 Profile 读写任意任务） |

### 1.3 任务生命周期（六状态状态机）

```
  triage ──→ todo ──→ ready ──→ running ──→ done ──→ archived
                              ↕                            ↑
                           blocked ───────────────────────┘
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| **triage** | 原始想法停车场 | 用户创建时加 `--triage`，或自动分解入口 |
| **todo** | 已创建但依赖未满足 | 有 parent link 且父任务未完成 |
| **ready** | 分配完毕等待认领 | 所有父任务 done，dispatcher 可认领 |
| **running** | Worker 正在执行 | Dispatcher 原子性 claim + spawn |
| **blocked** | 等待人工输入 / 熔断触发 | Worker 调用 `kanban_block` 或连续失败超限 |
| **done** | 完成 | Worker 调用 `kanban_complete(summary=...)` |
| **archived** | 已归档隐藏 | 用户手动归档 |

### 1.4 Worker 工具集（9 个专用工具）

| 工具 | 用途 | 调用方 |
|------|------|-------|
| `kanban_show` | 读当前任务全量上下文（含历史 run、评论） | Worker 启动后首调用 |
| `kanban_list` | 列出任务摘要（可按 assignee/status/tenant 过滤） | Orchestrator 发现任务 |
| `kanban_complete` | 完成任务并传递结构化 handoff | Worker 干完活 |
| `kanban_block` | 阻塞任务请求人工介入 | Worker 卡住时 |
| `kanban_heartbeat` | 发送心跳信号存活 | 长任务中（至少 1 次/小时）|
| `kanban_comment` | 向任务追加持久化评论 | Worker/Human 交流 |
| `kanban_create` | **仅 Orchestrator**：创建子任务 | 编排器拆分目标 |
| `kanban_link` | **仅 Orchestrator**：建立父子依赖 | 编排器定义流水线 |
| `kanban_unblock` | **仅 Orchestrator**：解除阻塞 | 人工处理后恢复 |

### 1.5 Dispatcher 调度器机制

Dispatcher 是运行在 **Gateway 内部**的长生命周期循环（默认 60s/tick），每轮执行：

1. **回收过期认领** — TTL 内未完成的 task reclaim
2. **回收崩溃 Worker** — PID 消失但 TTL 未到
3. **提升 ready** — 所有 parent done 的 todo → ready
4. **原子认领 + 派生** — 为 ready task 分配 worker 进程
5. **自动分解 triage** — LLM 将粗粒度 idea 扇出为子任务图

**关键配置项**：

```yaml
kanban:
  dispatch_in_gateway: true           # 默认: 在 Gateway 内运行
  dispatch_interval_seconds: 60       # 调度间隔
  failure_limit: 2                    # 连续失败 N 次后自动 block
  dispatch_stale_timeout_seconds: 14400 # 4 小时无心跳视为 stale
  auto_decompose: true                # 自动分解 triage 任务
  auto_decompose_per_tick: 3          # 每次 tick 最大分解数
```

### 1.6 熔断器 (Circuit Breaker)

连续失败超过 `failure_limit`（默认 2）次后，任务自动 **block** 并记录最后错误为原因。

**软阻止（不 increment failure 计数）**：
- `blocker_auth` — 上次失败是配额/认证/429 错误
- `recent_success` — 一小时内有成功完成的 run
- `active_pr` — 近期评论中发现 GitHub PR URL

### 1.7 结构化交接 (Structured Handoff)

这是多阶段工作流的**核心通道**——不是装饰品：

```python
kanban_complete(
    summary="implemented token bucket, keys on user_id with IP fallback",
    metadata={
        "changed_files": ["limiter.py", "tests/test_limiter.py"],
        "tests_run": 14,
        "dependencies": ["parent_task_id"],
        "retry_notes": "what failed before",
        "residual_risk": ["what needs human review"]
    },
    result="rate limiter shipped"
)
`

下游 Worker 通过 `kanban_show()` 获取的 `worker_context` 包含：
1. 该任务的**所有历史 run**（outcome、summary、error）
2. **每个父任务**的最新完成 run 的 summary 和 metadata

### 1.8 多看板 (Multi-Board) 隔离

一台机器可以运行多个完全隔离的任务板：

| 隔离维度 | 实现 |
|---------|------|
| 数据库 | `~/.hermes/kanban/boards/<slug>/kanban.db` |
| 工作空间 | 每个 board 独立的 `workspaces/` 目录 |
| 日志 | 每个 board 独立日志目录 |
| Worker 可见性 | Worker 只能看到自己 board 的 task（通过环境变量隔离） |

```bash
# 创建新看板
hermes kanban boards create atm10-server \
    --name "ATM10 Server" --icon 🎮 --switch

# 操作指定看板
hermes kanban --board atm10-server list
```

### 1.9 九大协作模式

| 编码 | 模式 | 形态 | 典型场景 |
|------|------|------|---------|
| P1 | Fan-out | N 个 sibling 同角色并行 | 研究 5 个角度 |
| P2 | Pipeline | 角色链式流转 | scout → editor → writer |
| P3 | Voting/quorum | N sibling + 1 aggregator | 3 研究员 → 1 审稿人 |
| P4 | Long-running journal | 同 profile + 共享 dir + cron | Obsidian 维护 |
| P5 | Human-in-the-loop | Worker block → human unblock | 歧义决策 |
| P6 | @mention | 正文中内联路由 | "@reviewer look at this" |
| P7 | Thread-scoped workspace | gateway thread 级隔离 | per-project 对话 |
| P8 | Fleet farming | 1 profile 管理 N subject | 50 社交账号 |
| P9 | Triage specifier | 一句话 → LLM 扩充为完整 spec | 需求细化 |

### 1.10 Dashboard Kanban UI 特性

- **六列拖拽看板**：triage / todo / ready / running / blocked / done
- **Profile Lane**：Running 列按 assignee 子分组（可切换平铺视图）
- **实时更新**：WebSocket 监听 `task_events` 表
- **侧边抽屉**：点击卡片展开详情、编辑、依赖管理、评论区、事件历史
- **Triage 列特殊操作**：
  - ⚗ **Decompose** — LLM 将任务扇出为子任务图
  - ✨ **Specify** — LLM 将一句话扩充为目标/方案/验收标准
- **REST API**：完整的 CRUD API（`/api/plugins/kanban/...`）

## 3. Profile 多实例机制

### 3.1 核心概念 (v0.6.0 引入)

Profile 让你在**同一台机器上运行多个完全独立的 Hermes 实例**，每个有自己的人格、记忆、技能和工作区。

```bash
# 创建新 Profile
hermes profile create work
hermes profile create personal
hermes profile create research

# 使用指定 Profile 启动
hermes -p work
hermes -p research

# 导出/分享/备份
hermes profile export work    # 输出 .tar.gz
hermes profile import work.tar.gz
```

### 3.2 每个 Profile 独立拥有的资源

| 资源 | 隔离范围 |
|------|---------|
| 配置 | 独立 config.yaml |
| 记忆库 | 独立 MEMORY.md / USER.md |
| 会话历史 | 独立 sessions/ 目录 |
| 技能集合 | 独立 skills/ 目录 |
| 工具权限 | 可按 Profile 限制不同 toolset |
| 网关服务 | 独立 PID、独立端口（自动冲突处理） |
| 凭证 | Token-lock 隔绝两个 Profile 用同一个 Bot 凭证 |

### 3.3 内存占用

每个 Profile 运行时内存 **< 500MB**（不含本地 LLM），意味着一台 16GB 内存的机器可以同时运行 **20+ 个 Profile**。

### 3.4 典型使用场景

| 场景 | Profile 配置示例 |
|------|-----------------|
| 工作 vs 个人隔离 | `work`(公司项目) / `personal`(私人事务) |
| 专岗专能 | `backend-dev`(写代码) / `reviewer`(审查) / `ops`(运维) |
| Kanban Worker | `researcher`(调研) / `writer`(写作) / `qa-dev`(测试) |
| 多租户 | 按 client 隔离，每个客户一个 Profile |

## 4. Mixture of Agents (MoA) 多模型协同推理

### 4.1 设计理念

MoA 不是多 Agent 协作——而是**多个外部 LLM 同时回答同一个问题，再聚合出最佳答案**。灵感来自 arXiv:2406.04692 论文。

```
用户问题
  ├── Claude Opus 4.6 (temp=0.6) ──┐
  ├── Gemini 3 Pro (temp=0.6) ──────┤
  ├── GPT-5.4 Pro (temp=0.6) ───────┤──→ Claude Opus 4.6 (temp=0.4) → 最终答案
  └── DeepSeek V3.2 (temp=0.6) ─────┘
        参考 Reference Models (多样性)     聚合 Aggregator (一致性)
```

### 4.2 默认配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 参考模型 | Claude Opus 4.6, Gemini 3 Pro, GPT-5.4 Pro, DeepSeek V3.2 | 4 个前沿模型并行 |
| 聚合模型 | Claude Opus 4.6 | 最强综合能力模型 |
| 参考温度 | 0.6 | 平衡创造力，保证多样性 |
| 聚合温度 | 0.4 | 专注综合，保证一致性 |
| 最少成功参考 | 1 | 只要 1 个成功即可聚合 |
| 每模型最大重试 | 6 次 | 弹性容错 |
| 总 API 调用 | 5 次 | 4 参考 + 1 聚合 |

### 4.3 与 Delegate Task 的区别

| 维度 | Delegate Task | Mixture of Agents |
|------|--------------|------------------|
| 目的 | 并行执行**不同任务** | 同一问题**多角度推理** |
| 隔离 | 完全对话隔离 | 只共享参考回复 |
| 模型 | 同模型或覆盖 | 4 参考 + 1 聚合 (5 个 API 调用) |
| 输出 | 每个任务独立摘要 | 单一综合答案 |
| 典型场景 | 研究/调试/多工作流 | 复杂数学/算法/高难度推理 |

### 4.4 适用场景

- 复杂数学证明
- 架构决策的多视角评估（技术/商业/伦理）
- 高可靠性需求的关键判断
- 需要消除单一模型偏见的场景

## 5. Observability 可观测性体系

### 5.1 为什么需要可观测？

原生 Hermes 的排障手段只有两类：
1. **零散日志** — 格式不统一，无法做趋势统计
2. **state.db** — 查询门槛高，缺少关键维度（工具耗时分布、Token 归因等）

核心痛点：
- **Token 成本缺乏归因**：哪天消耗暴涨？哪个 model/platform 异常？
- **性能瓶颈难区分**：慢在哪？模型延迟？工具执行？网络？
- **故障归因缺数据**：偶发抖动 vs 持续恶化？

### 5.2 四维可观测框架

| 维度 | 解决的问题 | 关键指标 |
|------|-----------|---------|
| **成本归因** | Token 流向追踪 | 按 model/provider/platform/host 拆解消耗 |
| **性能拆解** | 到底慢在哪一环 | P50/P90/P99 各阶段耗时分布 |
| **稳定性监控** | 哪个组件出故障 | 各工具/Provider 失败次数+失败率+趋势 |
| **链路追踪** | 这一次对话发生了什么 | Trace 全链路：每步 Token/耗时/状态/报错 |

### 5.3 方案选型

#### 方案 A：内置 Langfuse 插件

```bash
# 一键启用（交互式）
hermes tools → Langfuse Observability → Cloud or Self-Hosted

# 或手动
pip install langfuse
hermes plugins enable observability/langfuse
# 设置 .env:
# HERMES_LANGFUSE_PUBLIC_KEY=pk-lf-...
# HERMES_LANGFUSE_SECRET_KEY=sk-lf-...
# HERMES_LANGFUSE_BASE_URL=https://cloud.langfuse.com
```

**特性**：
- Fail-open 设计：无 SDK/无凭证/错误 → 静默 no-op，不影响 Agent 循环
- Hook 埋点：pre_api_request / post_api_request / pre_tool_call / post_tool_call
- Session 分组：按 Hermes session ID 聚合到 Langfuse session
- 大文件截断：read_file 超过 12000 字符自动 head+tail 摘要
- 可调参数：采样率、环境标签、版本标签、调试模式

#### 方案 B：hermes-otel 插件（社区）

```bash
hermes plugins install briancaffey/hermes-otel
~/git/hermes-agent/venv/bin/pip install opentelemetry-api opentelemetry-sdk opentelemetry-exporter-otlp-proto-http
```

**支持的 OTLP 后端**（10+）：

| 后端 | 类型 | 数据 |
|------|------|------|
| Phoenix (Arize) | 云/本地 | traces + metrics |
| Langfuse | 云/自建 | traces only |
| LangSmith | 云 | traces only |
| SigNoz | 云/自建 | traces + metrics + logs |
| Jaeger | 本地 | traces only |
| Grafana Tempo | 本地/云 | traces only |
| Grafana LGTM | 本地 | traces + metrics + logs |
| Uptrace | 自建 | traces + metrics + logs |
| OpenObserve | 自建 | traces + metrics + logs |
| 任何 OTLP HTTP endpoint | 自定义 | 取决于接收端 |

**核心优势**：
- **双协议属性**：同时 emit `gen_ai.*`(Langfuse) 和 `llm.token_count.*`(Phoenix)，UI 即开即用
- **多后端 fan-out**：同一条 span 并行发往 Phoenix + Langfuse + Jaeger，互不阻塞
- **零热路径延迟**：BatchSpanProcessor，span.end() 只是入队
- **隐私模式**：`capture_previews: false` 剥离输入输出预览，保留元数据流
- **孤儿 span 清扫**：TTL sweeper 清理长期 abandoned sessions

**Span 层级结构**：

```
session.{platform} / cron [root, GENERAL]
└── llm.{model} [LLM — input, output, total tokens]
    ├── api.{model} [LLM — prompt/completion tokens, duration]
    │   └── tool.{name} [TOOL — args, result, outcome]
    └── api.{model} [LLM — second round-trip]
```

每 turn root span 附带 summary：tool count、tool names、skills used、API call count、final status

#### 方案 C：云厂商集成

| 云厂商 | 方案 | 特点 |
|--------|------|------|
| **火山引擎 TLS** | hermes-tls 插件 | 一键部署脚本、AK/SK + API Key 双鉴权、四维看板 |
| **阿里云 SLS** | hermes-cms 插件 | CMS 2.0 AI 应用可观测、审计大盘自动生成、Trace 还原 |

### 5.4 Span 事件类型完整列表

**生命周期事件**：created / promoted / claimed / completed / blocked / unblocked / archived
**编辑事件**：assigned / edited / reprioritized / status
**Worker 遥测事件**：spawned / heartbeat / reclaimed / crashed / timed_out / stale / respawn_guarded / spawn_failed / protocol_violation / gave_up

## 6. Background Review 后台自省引擎

### 6.1 触发机制

Agent 在对话过程中维护**两个独立计数器**：

| 计数器 | 递增时机 | 触发阈值 | 默认值 |
|--------|---------|---------|-------|
| Memory 计数器 | 每轮对话 turn | 写 MEMORY.md | 10 |
| Skill 计数器 | 每次 tool call | 创建/改进 skill | 10 |

任一计数器达到阈值 → **fork 一个静默后台 Agent** 进行回顾。

### 6.2 Fork 机制细节

```python
def _spawn_background_review(messages_snapshot, review_memory, review_skills):
    fork = AIAgent(
        model=self.model,
        provider=self.provider,
        max_iterations=8,           # 轻量级最多 8 步
        quiet_mode=True,             # stdout → /dev/null
        conversation_history=messages_snapshot,  # 父对话快照
        skip_context_files=True,
    )
    # fork 共享父的 _memory_store 和 skill 目录
    # 可以调用 skill_manage(action='create/patch')
```

**关键设计**：
- **守护线程（非阻塞）**：不阻塞主对话
- **共享记忆写入**：review 结果直接生效
- **Review 自身的 nudge 触发禁用**：防止无限递归
- **限制 8 次 tool call**：控制 API 成本

### 6.3 三种 Review Prompt

| Prompt | 关注点 | 输出 |
|--------|-------|------|
| `_MEMORY_REVIEW_PROMPT` | 提取值得记住的事实 | → 写入 MEMORY.md |
| `_SKILL_REVIEW_PROMPT` | 提炼可复用的流程 | → 创建/改进 skill |
| `_COMBINED_REVIEW_PROMPT` | 同时做 memory + skill | → 双管齐下 |
