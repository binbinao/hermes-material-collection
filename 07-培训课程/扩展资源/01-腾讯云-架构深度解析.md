# Hermes Agent 技术架构深度解析：110K+ Star 自进化 AI Agent 架构设计

**2026年「术哥无界」系列实战文档第 X 篇原创，Hermes 最佳实战第 1 篇**

---

## 一、项目定位与核心差异化

Hermes Agent 是 **Nous Research 发布**的开源自进化 AI Agent，截至发稿 GitHub Star 数超 110K，贡献者超 240 人，提交次数超 4800 次，采用 MIT 协议开源，技术栈以 Python 3.11+ 为主，支持 OpenAI、Anthropic、Bedrock、OpenRouter 等 10+ 模型后端，切换模型仅需 `hermes model` 命令，无需修改代码。

### 核心差异化

与市面主流 Agent 框架聚焦"让 LLM 更好调用工具"不同，Hermes Agent 核心定位是 **self-improving AI agent（自进化 AI Agent）**，从架构原生层面实现闭环学习，而非事后补丁，核心包含 5 个环节：

1. **策划记忆**：任务完成后自主判断需保留的信息
2. **创建 Skill**：识别重复模式，自动生成 Markdown 格式 Skill 文件
3. **Skill 自改进**：现有 Skill 失败时自动优化
4. **FTS5 召回**：基于 SQLite 全文索引按需检索历史对话
5. **用户建模**：从用户行为推断偏好（Honcho 方言式建模）

整套机制对应认知科学三类记忆：**情景记忆（会话历史）、语义记忆（MEMORY.md 持久事实）、程序性记忆（Skill 文件）**。

## 二、核心架构概览

### 1. 核心目录职责

| 目录/文件 | 职责说明 |
|--------------|----------|
| `run_agent.py` | AIAgent 主类，Agent 循环核心，管理整个生命周期 |
| `tools/` | 40+ 工具实现（终端、浏览器、文件、搜索、MCP 等） |
| `toolsets.py` | Toolset 定义和组合系统 |
| `agent/prompt_builder.py` | System Prompt 组装器，支持缓存加速 |
| `agent/memory_manager.py` | 记忆管理，采用双 Provider 架构 |
| `agent/context_engine.py` | 上下文引擎，抽象基类，支持可插拔实现 |
| `agent/context_compressor.py` | 默认上下文压缩实现 |
| `hermes_state.py` | SQLite 状态存储，支持 FTS5 全文搜索 |
| `gateway/run.py` | 多平台消息网关，单一进程处理 17+ 平台消息 |
| `skills/` | 内置 Skill 库，覆盖 24 个分类 |
| `plugins/` | 插件系统 |
| `hermes_cli/` | CLI 接口，共 51 个模块 |
| `environments/` | 执行环境后端，支持 local/Docker/SSH/Daytona/Singularity/Modal 等 |

### 2. 关键架构设计决策

- **Agent 循环与工具执行同进程**：通过 `AIAgent` 类管理生命周期，未采用微服务架构，降低复杂度
- **工具自注册模式**：每个工具文件在模块级别调用 `registry.register()` 声明自身，无需中心化注册表
- **上下文压缩可插拔**：`ContextEngine` 为抽象基类，默认实现为 `ContextCompressor`，支持自定义替换
- **状态存储采用 SQLite + WAL 模式**：单文件数据库，支持多读者 + 单写者，适配网关多平台并发场景
- **MCP 双向支持**：既可作为 MCP 客户端连接外部工具服务器，也可作为 MCP 服务端被 Cursor、VS Code 等接入
- **Profile 系统（v0.6.0 引入）**：支持多实例 Profile，每个 Profile 拥有独立配置、记忆库、会话历史、Skill 集合和工具权限，互不干扰
- **低资源占用**：不跑本地 LLM 时，Agent 进程内存占用不到 500MB，可在 5 美元/月的 VPS 上运行

## 三、Agent 循环深度拆解

`run_agent.py` 中的 `AIAgent` 类是系统核心，管理对话流、工具执行、响应处理全流程，基本流程为：

`接收用户消息 → 构建请求（含 system prompt + 记忆 + 上下文） → 调用 LLM → 解析响应 → 若包含工具调用则执行工具 → 工具结果加入上下文 → 再次调用 LLM → 直到 LLM 无工具调用 → 返回最终响应`

### 1. 迭代预算控制

通过 `IterationBudget` 类实现线程安全迭代计数，避免 Agent 陷入死循环：
- 父 Agent 默认 90 次迭代
- 子 Agent 默认 50 次迭代

迭代预算耗尽则强制停止，防止 token 过度消耗

### 2. 并行工具执行

通过 `_should_parallelize_tool_batch()` 方法将工具分为三类，实现安全的并行执行：

| 分类 | 策略说明 | 典型工具 |
|------|----------|----------|
| `_NEVER_PARALLEL_TOOLS` | 永远不并行 | `clarify`（需要用户交互） |
| `_PARALLEL_SAFE_TOOLS` | 只读安全，可并行 | `web_search`、`read_file` |
| `_PATH_SCOPED_TOOLS` | 路径隔离，满足条件可并行 | `read_file`、`write_file`、`patch` |

最大并发工作线程数硬编码为 8（`_MAX_TOOL_WORKERS`），避免资源冲突

### 3. 中断与转向机制

通过 `_interrupt_requested` + `_pending_steer` 双标志位实现用户指令响应：
- 不打断当前正在执行的工具，等工具批次完成后再处理新指令
- 将用户新需求作为 `_pending_steer` 注入下一轮对话，保证操作原子性，避免文件写入一半等异常

### 4. 多 API 模式支持

支持 `chat_completions`、`codex_responses`、`anthropic_messages`、`bedrock_converse` 四种 API 模式，模型提供商路由自动检测，模型切换无需修改业务代码，需切换模型时创建新 Agent 实例即可

### 5. PromptBuilder 组件

负责组装完整 system prompt，包含 `DEFAULT_AGENT_IDENTITY`（核心身份定义）、平台特定提示、技能索引、上下文文件、记忆快照等多个层次，同时实现两级缓存（LRU 内存缓存 + 磁盘快照持久化）加速 Skill 索引读取，降低启动耗时

## 四、工具系统：自注册 + Toolset 组合

### 1. ToolRegistry 自注册模式

核心实现在 `tools/registry.py`，采用优雅的自注册设计：

- 每个工具文件在模块级别调用 `registry.register()` 声明自身的 schema、handler、toolset 归属等信息，注册表用 `threading.RLock()` 保证线程安全
- 通过 `discover_builtin_tools()` 方法基于 AST 分析自动检测包含 `registry.register()` 调用的 `.py` 文件，无需手动维护工具列表
- 支持动态注册/注销，适配 MCP 服务器刷新场景：MCP 工具上线时调用 `register()`，下线时调用 `deregister()`

`ToolEntry` 核心结构包含：工具名、所属工具集、JSON Schema、处理函数、检查函数、是否需要执行环境、是否异步、描述、显示图标

### 2. Toolset 组合系统

`toolsets.py` 实现组合式工具集系统，Toolset 之间可以互相 `includes`，支持递归解析：

- **核心工具列表** `_HERMES_CORE_TOOLS` 包含 63 个工具，所有平台共享，覆盖 Web、终端、文件、视觉、技能、浏览器、规划、代码执行、定时任务等类别
- 支持 20+ 平台特定 Toolset（如 `hermes-cli`、`hermes-telegram`、`hermes-discord` 等），`hermes-gateway` 是所有平台工具的并集
- 新接入平台时仅需定义平台特有工具，再 includes 核心工具集即可，避免重复定义，同时支持通过 MCP 协议接入 6000+ 外部应用

## 五、记忆与学习闭环

这是 Hermes Agent 与同类框架的核心差异点，实现完整的自进化能力

### 1. 双文件记忆架构

`agent/memory_manager.py` 的 `MemoryManager` 采用双 Provider 架构：1个内置 Provider（管理 MEMORY.md 和 USER.md）+ 最多1个外部 Provider（如 Honcho），两个文件分工明确：

- **MEMORY.md**：Agent 个人笔记，记录环境事实、项目约定、工具使用技巧
- **USER.md**：Agent 对用户的认知，记录用户偏好、沟通风格、期望

记忆注入时使用 `<memory-context>` 标签做上下文围栏，防止模型误将记忆上下文当作新用户输入，同时内置安全扫描机制，检测记忆内容中的恶意注入指令。记忆条目之间用 `§`（section sign）分隔，降低与正文内容冲突的概率

### 2. 冻结快照模式

为解决系统提示稳定性问题，采用读写分离设计：

- 会话开始时，将当前记忆内容作为快照注入系统提示，整个会话期间系统提示保持不变
- 中途写入的记忆仅更新磁盘文件，不刷新系统提示，保证前缀缓存（KV cache）有效，提升性能
- 代价是会话内对记忆的修改需等到下一个会话才能被读取，多数场景下该延迟可接受

### 3. Skill 系统

Skill 文件存放在 `~/.hermes/skills/` 目录下，为标准 Markdown 文件，创建和改进流程为：

1. Agent 完成复杂任务后，自动识别可复用模式
2. 生成 Markdown 格式 Skill 文件，包含触发条件、执行步骤、注意事项
3. 使用 Skill 时发现过时、报错等问题，立即 patch 更新

`agent/prompt_builder.py` 中的 `SKILLS_GUIDANCE` 提供 Skill 创建和更新指导规则，同时实现两级缓存加速 Skill 索引读取

### 4. 会话搜索

`hermes_state.py` 的 `SessionDB` 基于 SQLite FTS5 实现跨所有会话消息的全文搜索，流程为：FTS5 召回相关片段 → LLM 总结 → 注入当前上下文。`prompt_builder.py` 中的 `SESSION_SEARCH_GUIDANCE` 指导 Agent 何时及如何使用会话搜索

**完整记忆召回链路**：用户提问 → Agent 判断是否需要历史信息 → 调用 `session_search`（FTS5 召回） → LLM 总结召回结果 → 结合 MEMORY.md/USER.md 快照 → 形成完整记忆上下文，实现多级检索

## 六、上下文管理：可插拔压缩引擎

### 1. ContextEngine 抽象架构

`agent/context_engine.py` 定义抽象基类 `ContextEngine`，`agent/context_compressor.py` 提供默认实现 `ContextCompressor`，可插拔设计支持自定义上下文管理策略（如 RAG 检索增强、基于重要性评分的选择性保留等），适配编程、对话、数据分析等不同场景的上下文处理需求

### 2. 压缩策略详解

`ContextCompressor` 核心参数：

| 参数 | 默认值 | 作用说明 |
|------|--------|----------|
| `threshold_percent` | 0.75 | 上下文使用率超过 75% 时触发压缩 |
| `protect_first_n` | 3 | 保护前 3 条消息（通常是 system prompt + 用户初始请求） |
| `protect_last_n` | 6 | 保护后 6 条消息（最近的对话上下文） |

**压缩具体流程**：
1. 保护头部和尾部消息不动
2. 中间部分调用辅助模型生成结构化摘要，包含已解决问题、待处理事项、活跃任务
3. 对工具返回的长文本做前置过滤，减少总结 token 消耗
4. 中间部分每段对话按比例分配总结 token 预算

另外 `trajectory_compressor.py` 提供轨迹压缩能力，主要用于科研场景的批量轨迹生成

### 3. PromptBuilder 安全扫描

`agent/prompt_builder.py` 内置 `_scan_context_content()` 方法，检测注入攻击。上下文文件（`.hermes.md`、`AGENTS.md`、`CLAUDE.md`、`.cursorrules`）优先级为：`.hermes.md > AGENTS.md > CLAUDE.md > .cursorrules`，扫描会检查这些文件中是否包含恶意指令，防止 prompt injection

## 七、子 Agent 委托机制

`tools/delegate_tool.py` 实现子 Agent 委托，允许主 Agent 生成独立子 Agent 实例并行处理任务

### 1. 隔离设计

子 Agent 实现严格隔离：
- 无父 Agent 对话历史访问权限
- 拥有独立终端环境
- 被禁止调用递归委托、用户交互类、记忆写入类等工具，避免污染主 Agent 状态

### 2. 并行与深度控制

- 默认最多 3 个并行子 Agent（`MAX_CONCURRENT = 3`），避免资源爆炸
- 委托深度默认 `MAX_DEPTH = 1`，子 Agent 不可再委托（避免递归爆炸）
- 支持 `leaf`（叶节点，仅执行任务）和 `orchestrator`（编排者，可再次委托）两种角色模式

## 八、多平台网关架构

`gateway/run.py` 实现统一消息网关，单一进程处理所有平台消息，支持 17+ 平台：Telegram、Discord、Slack、WhatsApp、Signal、Email、SMS、Home Assistant、Mattermost、Matrix、钉钉、飞书、微信、企业微信、QQ、BlueBubbles、Webhook，v0.9.0（代号 "the everywhere release"）额外支持 Termux（Android）和 iMessage

### 1. Agent 缓存策略

网关采用 LRU 缓存 + 空闲 TTL 淘汰策略管理 Agent 实例：
- 缓存容量：128 个 Agent 实例
- 空闲淘汰：1 小时无活动自动清理，平衡内存使用和响应速度

### 2. 协作机制

网关收到消息后，从缓存查找或创建对应 Agent 实例，将消息交给 Agent 主循环处理，Agent 响应再通过网关发回对应平台。不同平台的消息格式差异（如 Telegram Markdown V2、Discord Embed、Slack Block Kit）在网关层统一处理，Agent 核心无需关心细节，支持跨平台继续同一对话

## 九、状态持久化：SQLite + FTS5

`hermes_state.py` 的 `SessionDB` 是整个系统的状态层，采用本地优先设计，所有数据存储在 `~/.hermes/` 目录，无云端同步，保护用户隐私

### 1. 技术选型

- **SQLite + WAL 模式**：支持多读者 + 单写者，适配网关多平台并发场景
- **FTS5 全文搜索**：支持跨所有会话消息的快速文本搜索
- **Schema 版本控制**：当前为 v8，支持自动迁移

### 2. 写竞争处理

针对 SQLite 高并发写入锁竞争问题，采用**随机抖动重试**策略：15 次重试，每次等待 20-150ms 随机时间，避免多个写请求同时重试导致的护航效应（convoy effect）

### 3. 会话管理

- `get_compression_tip()` 方法沿压缩-续链前进，追踪会话完整历史
- 支持会话标题自动生成和唯一性约束，族系命名如 "title #2"、"title #3"

## 十、架构总结

### 核心优势

1. **闭环学习原生支持**：从 `PromptBuilder` 的 `MEMORY_GUIDANCE` 到 `MemoryManager` 双文件架构，再到 `SessionDB` 的 FTS5 搜索，整套学习闭环在架构层面贯通，而非事后补丁
2. **工具系统灵活可扩展**：自注册 + Toolset 组合设计，新平台接入成本低
3. **上下文管理可定制**：可插拔压缩引擎为不同场景优化留下空间
4. **生产级可靠性设计**：迭代预算、中断机制、并发控制等设计，体现对 Agent 实际运行问题的深入理解
5. 完整实现了 Mitchell Hashimoto 提出的 **Harness Engineering 五大组件**（指令层、约束层、反馈层、记忆层、编排层），是目前开源 Agent 框架中完成度较高的实现

### 待优化局限

1. 子 Agent 并发上限仅 3 个，复杂并行场景受限
2. 项目迭代节奏快，不到 3 周发布 4 个大版本，稳定性需持续观察
3. 不支持 Windows 原生运行，必须通过 WSL2
4. 存在架构级抄袭争议（中国团队指控），暂未得出最终结论

## 十一、行业价值

从架构设计角度看，Hermes Agent 的闭环学习和上下文管理方案，对 Agent 相关架构设计有较高参考价值，是当前开源生态中完成度较高的自进化 Agent 实现。

**适用场景**：自动化软件工程任务（代码审查、重构、CI/CD 流水线、复杂多步构建流程）

**不适用场景**：需要高度并行子 Agent 协作的超大规模任务、Windows 原生环境、对稳定性要求极高的生产环境（因迭代节奏过快）
